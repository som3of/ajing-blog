⚡ 批量侦察怎么做快：从串行扫描到分层并发

2026-03-13 · 方法论 / 工程化

为什么批量任务总是慢？

很多人做侦察时，习惯拿到一个域名后直接从头扫到尾：子域名、目录、POC、敏感文件，一个目标做完再做下一个。这种方式在目标很少时问题不大，但一旦变成几十、几百甚至上千个资产，吞吐量会迅速崩掉。

根因通常有三个：

  • 串行思维：把可以并发的事情串起来做。
  • 没有筛选：不区分高价值和低价值目标，导致无差别重扫。
  • 输出太晚：非要等全跑完才汇总，结果过程不可见，失败也难定位。

一个更实用的思路：分层扫描

我现在更认可的方式是把侦察拆成几个层次，每一层只解决一个问题：

  1. Layer 0 资产收集：先把目标面尽量收全。
  2. Layer 1 快速筛选：迅速找出活着的、值得继续打的目标。
  3. Layer 2 定向扫描：根据类型做目录、POC、敏感路径检测。
  4. Layer 3 结果聚合：把发现集中起来做去重和分类。
  5. Layer 4 报告输出：把可交付内容尽快产出。
重点：先做“筛选”，再做“重活”。这样扫描速度和结果质量都会更稳定。

怎么把速度提起来？

1. 让轻任务先跑

子域名枚举、端口探测、HTTP 存活检测，这些都应该优先并发完成。因为它们的目的不是挖深，而是把无效目标尽快淘汰掉。

# 例子：先把“活着的 Web 资产”筛出来
subfinder -d target.com -silent | httpx -threads 100 -silent -o alive.txt

2. 指纹之后再分流

不是所有目标都该走同一套扫描策略。比如识别到 Confluence、Jenkins、WordPress、Tomcat 后，就应该切换到更贴合的字典和模板，而不是继续一锅烩。

3. 控制重扫描范围

目录爆破和 POC 扫描最吃时间，应该只打在 Layer 1 里已经证明“活着且有价值”的目标上，而不是对全量资产直接开火。

4. 边跑边输出

批量任务不应该等全结束再看结果。最好的状态是：一边扫描，一边沉淀中间产物，例如 alive.txt、fingerprints.txt、sensitive.txt。

高并发不等于盲目开大线程

很多时候任务慢，不是线程数不够，而是资源调度混乱。真正有效的做法是按任务性质分别配置：

  • HTTP 存活检测：高并发、短超时
  • 目录扫描：中等并发、按指纹选字典
  • POC 扫描：按风险优先级跑 critical/high
  • 聚合报告：串行即可，但要快
CONFIG = {
  'http_threads': 100,
  'dir_threads': 50,
  'poc_threads': 30,
  'http_timeout': 3,
  'batch_size': 1000
}

我现在更在意的，不只是“快”

批量侦察真正难的不是跑工具，而是把流程工程化。一个能长期复用的流程，至少要满足：

  • 可重复：同样输入，结果路径稳定。
  • 可观察：随时知道跑到哪一层、卡在哪一步。
  • 可中断：哪怕中途停掉，也保留中间结果。
  • 可交付：最后不是一堆原始输出，而是整理过的结论。

结语

批量任务做不快，通常不是工具不够强，而是流程没有分层。把筛选、分流、重扫描、聚合拆开以后，效率会明显提升，结果也更容易复盘。

我现在的目标不是“扫得像在表演”，而是把侦察做成一个稳定、可复用、可交付的系统。
